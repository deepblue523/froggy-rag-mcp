# Froggy on RAG v1.2.4 — Re-enable permissive CORS for inbound passthrough listeners

**Release tag:** `v1.2.4` (set when publishing)  
**Package version:** 1.2.4 (bump with `npm run version:patch` before tag/build)  
**Type:** Patch release (browser compatibility / integration restore)

**Baseline:** Changes since **v1.2.2** only. For prior context, see **`release-notes/v1.2.2.md`**.

---

## Summary

This release re-enables permissive CORS behavior on inbound LLM passthrough listeners (both Ollama-compatible and OpenAI-compatible routes). The listeners now always emit permissive `Access-Control-Allow-*` headers and handle browser preflight `OPTIONS` requests with `204`, restoring direct browser compatibility for cross-origin local integrations.

## Highlights

### Inbound passthrough CORS restore

- **`src/main/services/passthrough-inbound-server.js`** — added **`applyPermissiveCors(app)`** middleware and registered it in both inbound app builders:
  - **`_ollamaApp()`**
  - **`_openAiApp()`**
- Middleware now always sends:
  - **`Access-Control-Allow-Origin: *`**
  - **`Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS`**
  - **`Access-Control-Allow-Headers: Content-Type, Authorization, X-Froggy-Namespace, x-froggy-namespace`**
- `OPTIONS` requests are short-circuited with **`204`** to satisfy browser preflight checks.

## Upgrade notes

- **Client impact:** Browser-based clients can again call inbound passthrough listener ports cross-origin without requiring an intermediate same-origin proxy.
- **Security note:** This intentionally broadens cross-origin access for inbound listeners. Use host/network controls if these ports should remain local-only.
- **Behavioral consistency:** MCP REST CORS behavior remains permissive; this patch aligns inbound passthrough behavior with that integration model.
