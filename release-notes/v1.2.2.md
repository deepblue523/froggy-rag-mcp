# Froggy on RAG v1.2.2 — Disable CORS on inbound passthrough listeners

**Release tag:** `v1.2.2` (set when publishing)  
**Package version:** 1.2.2 (bump with `npm run version:patch` before tag/build)  
**Type:** Patch release (security hardening / network surface tightening)

**Baseline:** Changes since **v1.2.1** only. For earlier context, see **`release-notes/v1.2.1.md`**.

---

## Summary

This release removes CORS handling from the inbound LLM passthrough HTTP listeners (both Ollama-compatible and OpenAI-compatible routes). The listeners no longer emit permissive cross-origin headers and no longer short-circuit browser preflight `OPTIONS` requests with a generic `204` response.

## Highlights

### Inbound passthrough CORS removal

- **`src/main/services/passthrough-inbound-server.js`** — removed the dedicated **`corsMiddleware`** implementation.
- Removed middleware registration from both inbound apps:
  - **`_ollamaApp()`**
  - **`_openAiApp()`**
- Result: no **`Access-Control-Allow-*`** headers are added by the inbound passthrough server.

## Upgrade notes

- **Client impact:** Browser-based clients that relied on permissive CORS for direct cross-origin calls to inbound passthrough ports will now be blocked by browser CORS policy unless routed through a same-origin backend/proxy.
- **Non-browser clients:** CLI tools, local SDKs, and server-to-server callers are unaffected by browser CORS enforcement and should continue to work as before.
- **Behavioral change:** Unhandled `OPTIONS` preflight behavior now follows normal Express routing behavior for the configured endpoints.
