# Froggy RAG MCP v1.0.15 — LLM Passthrough via inbound HTTP and per-provider settings

**Release tag:** `v1.0.15`  
**Package version:** 1.0.15  
**Type:** Feature refactor (MCP surface change) and UX fixes

---

## Summary

This release **narrows the MCP tool surface** for RAG-plus-LLM: the **`llm_rag_passthrough`** tool is removed in favor of **`search_vector_store`** (and **`list_namespaces`**) on MCP, while **RAG + your upstream model** is exercised through the app’s **local inbound HTTP** listeners (Ollama-style and OpenAI-compatible) when **LLM Passthrough** is enabled under **Settings → Server**. LLM endpoint configuration is split into **per-provider** fields (with automatic migration from the older single base URL / model / API key), and the **splash screen** startup race with the main window is fixed.

## Highlights

- **MCP:** `tools/list` no longer advertises `llm_rag_passthrough`. Server instructions now describe using the desktop app (**Settings → Server**: LLM Passthrough + inbound ports) for chat completions that inject retrieval, not an MCP tool.
- **Per-provider LLM settings** — separate base URL, model, and API key for **Ollama** vs **OpenAI**-compatible upstreams, with resolution that falls back to legacy `llmPassthroughBaseUrl` / `llmPassthroughModel` / `llmPassthroughApiKey` when the new keys were never saved. One-time migration copies legacy values into the correct provider slot on load.
- **Single master gate for inbound HTTP** — inbound passthrough listeners start only when **`llmPassthroughEnabled`** is on; the older **`passthroughListenEnabled`** flag is kept in sync on read/save for existing `settings.json` files.
- **UI (Settings → Server)** — clearer LLM Passthrough and inbound listener controls; switching **provider** preserves separate draft values per provider. The in-app **test send** calls the **local inbound** chat endpoint (with namespace header when applicable) instead of a dedicated IPC “chat” path, with persisted retrieval algorithm / timeout for that panel.
- **CLI** — removed `passthrough` / `llm_passthrough` aliases and the `llm_rag_passthrough` usage example (see `scripts/mcp-example-requests.http` for the updated REST/MCP flow comments).
- **Startup** — splash is shown on **`did-finish-load`** (and a short fallback path) so it is not torn down before **`show`**, avoiding a blank flash when the main window wins the **ready-to-show** race.

## Upgrade notes

- **If you used `llm_rag_passthrough` from MCP or the CLI:** switch to **`search_vector_store`** (and your own model call), **or** enable **LLM Passthrough** and an **inbound** listener in the app and point your client at that local **Ollama**- or **OpenAI**-compatible URL instead.
- After upgrade, confirm **Settings → Server** has **LLM Passthrough** enabled if you expect inbound listeners; legacy installs that only toggled the old “listen” flag are aligned to **`llmPassthroughEnabled`** in stored settings where applicable.
